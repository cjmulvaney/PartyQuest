# Party Quest v2.6 ŌĆö Bug Fixes Implementation Plan

## Context
Code review identified 15 bugs across the Party Quest app (P0-P3), including wide-open RLS policies, race conditions, silent error swallowing, and inconsistent mission assignment logic. This plan addresses all 15 issues from `v2.6-bug-fixes-and-updates.md`.

**Key architectural decision:** Participant write operations will move to `SECURITY DEFINER` RPC functions in Postgres, since participants don't use Supabase Auth. This secures the API while also solving race conditions atomically.

---

## Phase 1: Database Migration
**New file:** `party-quest/supabase/v2.6-migration.sql`

### 1A: Helper function ŌĆö `generate_access_code(length)`
- Postgres function to generate random access codes (same charset as frontend `generateCode`)
- Used by RPC functions below

### 1B: RPC ŌĆö `rpc_register_participant(p_event_code, p_name)` 
Solves issues **#1, #3, #7, #9** in one atomic function:
- Validates event exists and is active
- Checks duplicate name via `LOWER(TRIM())` match (issue #9)
- Atomic max_participants check-and-insert (issue #3)
- Access code generation with retry on unique violation (issue #7)
- Returns the new participant record as JSON
- `SECURITY DEFINER` so it bypasses RLS

### 1C: RPC ŌĆö `rpc_join_event(p_event_code, p_access_code)`
Solves issue **#11** (joined_at error handling) at DB level:
- Validates event + access code
- Sets `joined_at` if null
- Returns participant record

### 1D: RPC ŌĆö `rpc_complete_mission(p_access_code, p_mission_id, p_notes, p_photo_url)`
Secures mission completion (issue #1):
- Validates access code ŌåÆ participant ŌåÆ mission ownership
- Updates only `completed`, `notes`, `photo_url`, `completed_at`

### 1E: Tighten RLS policies (issue #1)
- Drop permissive INSERT/UPDATE policies on `participants` and `participant_missions`
- Add organizer-scoped policies (via `auth.uid()` ŌåÆ `events.organizer_id` join)
- Keep SELECT as-is (needed for leaderboard/feed)
- All participant writes now go through RPCs

### 1F: Partial unique index for names (issue #9)
```sql
CREATE UNIQUE INDEX idx_unique_active_name_per_event
  ON participants (event_id, LOWER(TRIM(name)))
  WHERE is_active = true;
```

## Deployment Notes

**Important:** The RLS policy changes and RPC functions (Phase 1) must be deployed to Supabase **before** the frontend changes (Phase 4) go live. Otherwise participant registration/join/completion will break because:
1. Old frontend does direct table inserts ŌåÆ new RLS blocks them
2. New frontend calls RPCs ŌåÆ RPCs don't exist yet

**Recommended deploy order:**
1. Run `v2.6-migration.sql` in Supabase SQL Editor
2. Deploy frontend (Phases 2-4 changes)

Phase 3 changes (simple error handling fixes) are safe to deploy independently at any time.
