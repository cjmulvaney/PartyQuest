# Party Quest v2.6 — Bug Fixes & Updates

**Date:** 2026-04-06
**Current Version:** 2.5.0
**Scope:** Code review findings across participant joining, self-registration, organizer event creation, and database security.

---

## P0 — Critical

### 1. RLS policies are wide open for participants and missions
**Files:** `supabase/schema.sql:219-242`

The participant and participant_missions tables have fully permissive RLS policies. Anyone with the Supabase anon key (exposed in the frontend bundle) can bypass the UI and hit the API directly to:

- **Read** all participants across every event, including names and access codes
- **Insert** fake participants into any event
- **Update** any participant record (name, access code, `joined_at`)
- **Mark any mission as completed**, change photos/notes, or modify unlock times

```sql
-- Current (too permissive)
create policy "Participants are publicly readable" on participants for select using (true);
create policy "Anyone can join as participant" on participants for insert with check (true);
create policy "Participants can update their own record" on participants for update using (true);

create policy "Participant missions can be updated" on participant_missions for update using (true);
```

**Solution:** Scope SELECT to participants within the same event (matched by access code or session). Restrict UPDATE on participants so only the record owner can modify it. Restrict UPDATE on participant_missions to only `completed`, `notes`, `photo_url`, `completed_at` columns for the participant's own missions. For INSERT, validate via a Postgres function that enforces event existence and status checks at the DB level.

### 2. `event_type` casing mismatch — "Bachelorette/Bachelor" breaks DB constraint
**Files:** `src/pages/organizer/CreateEvent.jsx:378` vs `supabase/schema.sql:36`

The UI option `'Bachelorette/Bachelor'` lowercases to `'bachelorette/bachelor'`, but the DB constraint only allows `'bachelorette'`:

```js
// CreateEvent.jsx:378
event_type: eventType.toLowerCase()  // => 'bachelorette/bachelor'
```

```sql
-- schema.sql:36
check (event_type in ('birthday', 'house party', 'bachelorette', 'work offsite', 'weekend trip', 'other'))
```

Selecting "Bachelorette/Bachelor" as event type will cause a Postgres check constraint violation and the event creation will fail.

**Solution:** Either update the DB constraint to include `'bachelorette/bachelor'`, or add a mapping in the frontend before inserting (e.g., `'Bachelorette/Bachelor' -> 'bachelorette'`).

## P1 — High

### 3. Race condition: Max participants check is not atomic with insert
**File:** `src/pages/Register.jsx:88-115`

The count check and insert are two separate queries. Under concurrent self-registration, multiple users can pass the check simultaneously and exceed the `max_participants` limit.

```js
// Line 88-100: CHECK
if (event.max_participants) {
  const { count } = await supabase
    .from('participants')
    .select('id', { count: 'exact', head: true })
    .eq('event_id', event.id)
    .eq('is_active', true)

  if (count >= event.max_participants) { ... return }
}

// Line 105-115: INSERT (not atomic with the check above)
const { data: participant } = await supabase
  .from('participants')
  .insert({ ... })
```

**Solution:** Use a Postgres RPC function that checks the count and inserts within a single transaction, or add a database trigger that enforces the constraint atomically.

### 4. Event config update error is swallowed in edit mode
**File:** `src/pages/organizer/CreateEvent.jsx:413-422`

When editing an existing event, the `event_config` update fires but the error is never checked. The event is marked as "launched" on line 424 regardless of whether the config actually saved.

```js
// No error handling — result is discarded
await supabase
  .from('event_config')
  .update({
    mission_count: missionCount,
    unlock_type: unlockType,
    unlock_schedule: schedule,
    tag_filters: selectedTags,
  })
  .eq('event_id', editEventId)

// Immediately marked as success
setCreatedEventId(eventData.id)
setLaunched(true)
```

If the update fails, the organizer sees "success" but the game config (mission count, unlock type, categories) remains stale.

**Solution:** Capture the error and throw it, same pattern as the event insert on line 404.

### 5. Bulk mission insert error not checked on event launch
**File:** `src/pages/organizer/CreateEvent.jsx:587-589`

```js
if (allRows.length > 0) {
  await supabase.from('participant_missions').insert(allRows)
  // No error check — could be 50+ rows failing silently
}
```

If this insert fails (duplicate mission-participant pair, network timeout on large batch), the event is still marked as launched on line 490. All participants would have zero missions.

**Solution:** Check the error. If it fails, don't set `launched = true`. Show an error prompting the organizer to retry.

### 6. Silent mission assignment failure on self-registration
**File:** `src/pages/Register.jsx:127-130`

```js
if (config) {
  await assignMissions(participant, config)  // No error handling
}
setSuccess({ accessCode, participantId: participant.id })
```

If `assignMissions` throws or the inner insert fails, the participant is created successfully but has zero missions. They'll see an empty Play screen with nothing to do.

Same issue in `src/pages/organizer/EventDetail.jsx:388-389` when organizer adds a participant mid-event.

**Solution:** Wrap in try/catch. If it fails, either retry or show a specific error. At minimum, warn the organizer that manual mission assignment may be needed.

## P2 — Medium

### 7. No retry on access code collision
**Files:** `src/pages/Register.jsx:103`, `src/pages/organizer/EventDetail.jsx:373`, `src/pages/organizer/CreateEvent.jsx:334`

`generateCode(6)` produces a 6-character code from a 31-char alphabet (~887M combinations). The code is inserted directly — if the DB `UNIQUE` constraint fires, the user sees a generic error with no retry.

```js
const accessCode = generateCode(6)
// Inserted directly — unique violation falls to catch with generic message
```

**Solution:** Wrap insertion in a retry loop (2-3 attempts) that catches Postgres error code `23505` (unique_violation) and regenerates the code before retrying.

### 8. Unlock time schedule index offset bug
**File:** `src/pages/organizer/CreateEvent.jsx:574-583`

```js
selected.forEach((m, i) => {
  allRows.push({
    unlock_time:
      unlockType === 'timed' && schedule && schedule[i]
        ? schedule[i]
        : null,
  })
})
```

`schedule` has `missionCount` slots. But `selected` is `sorted.slice(0, needed)` where `needed = missionCount - alreadyAssigned.size`. If a participant already has 1 mission and needs 2 more, `i` will be 0 and 1, mapping to `schedule[0]` and `schedule[1]`. But the correct unlock times for their 2nd and 3rd missions should be `schedule[1]` and `schedule[2]`.

**Solution:** Use `alreadyAssigned.size + i` as the schedule index instead of `i`.

### 9. Race condition: Duplicate name check is not atomic
**File:** `src/pages/Register.jsx:74-86`

The `ilike` name check and the `insert` are separate queries. Two users submitting "John" at the same time can both pass the check and both get inserted.

```js
const { data: existingName } = await supabase
  .from('participants')
  .select('id')
  .eq('event_id', event.id)
  .ilike('name', name.trim())
  .eq('is_active', true)
  .limit(1)

// ... later, insert happens separately
```

**Solution:** Add a partial unique index at the database level:
```sql
CREATE UNIQUE INDEX idx_unique_active_participant_name
  ON participants (event_id, lower(name))
  WHERE is_active = true;
```
Then handle the unique violation error in the frontend with a user-friendly message.

### 10. Inconsistent mission assignment algorithms across files
**File:** `src/pages/organizer/EventDetail.jsx:827` vs `Register.jsx:195` and `CreateEvent.jsx:566`

EventDetail uses a simple random shuffle when an organizer adds a participant mid-event:
```js
// EventDetail.jsx — random shuffle
const shuffled = [...missions].sort(() => Math.random() - 0.5)
const selected = shuffled.slice(0, eventConfig.mission_count)
```

Register.jsx and CreateEvent.jsx use leveled round-robin (least-assigned first). This means participants added by the organizer via the dashboard get random missions, while self-registered participants get balanced ones. Over time this leads to uneven mission distribution.

**Solution:** Use the same leveled round-robin algorithm in all three locations. Ideally extract to a shared utility function in `src/lib/`.

### 11. `joined_at` update has no error handling
**File:** `src/pages/Join.jsx:60-65`

```js
if (!participant.joined_at) {
  await supabase
    .from('participants')
    .update({ joined_at: new Date().toISOString() })
    .eq('id', participant.id)
  // Error silently ignored — user navigates to /play regardless
}
```

If this update fails, the organizer's dashboard will show the participant as "not yet joined" even though they're actively playing. Causes data inconsistency in event analytics.

**Solution:** At minimum, log the error. Optionally show a non-blocking warning toast.

## P3 — Low / Cleanup

### 12. Dead code: Early self-registrant query on new event
**File:** `src/pages/organizer/CreateEvent.jsx:473-481`

```js
// Queries for self-registered participants on a brand-new event (just inserted)
const { data: earlyRegistrants } = await supabase
  .from('participants')
  .select('*')
  .eq('event_id', eventData.id)  // This event was literally just created
  .eq('source', 'self')
```

This will always return empty since no one could have self-registered before the event existed. Adds a wasted DB query per event creation.

**Solution:** Remove this query block, or move it to only run in edit mode where it would actually make sense.

### 13. `mission_count` DB constraint vs UI mismatch
**File:** `supabase/schema.sql:66`

```sql
mission_count integer default 3 check (mission_count between 1 and 5)
```

If the UI allows values outside 1-5 (via direct input or a slider without bounds), the DB insert will fail with a check constraint error. Verify the UI enforces the same 1-5 range.

**Solution:** Add `min={1} max={5}` to the mission count input in the UI, or update the DB constraint if higher counts should be supported.

---

### 14. Clipboard copy shows "Copied!" even on failure
**Files:** `src/pages/Register.jsx:17`, `src/pages/organizer/CreateEvent.jsx:593`

```js
navigator.clipboard?.writeText(text)  // Optional chaining — won't throw, but also won't copy
setCopyToast(label || 'Copied!')       // Always shows success
```

The Clipboard API can silently fail in non-HTTPS contexts or if permissions are denied. The optional chaining prevents a crash, but the toast always shows "Copied!" even when nothing was actually copied.

**Solution:** Use `await navigator.clipboard.writeText(text)` inside a try/catch, and only show the success toast in the success path.

---

### 15. `useEffect` dependency array missing `startTime`
**File:** `src/pages/organizer/CreateEvent.jsx:79-88`

```js
useEffect(() => {
  if (!startTime) {
    // set defaults...
  }
}, [])  // Empty deps — references startTime but doesn't list it
```

Works correctly on mount since `startTime` is initially `''`, but violates the exhaustive-deps rule. Not a functional bug currently, but could cause issues if the component logic changes.

**Solution:** Add `startTime` to the dependency array, or suppress the lint warning with a comment explaining the intent.
